Install Sample Application on Tenant Cluster

In this section, you will be installing basic Node.js application on Tenant cluster. You will also be testing the high availability and resilience of deployed Node.js application.

Prerequisites

  1. Ensure OpenShift Data Foundation operator is installed.

  2. Ensure Image Registry operator is using Noobaa storage with Red Hat OpenShift Data Foundation.

    Sample output:
    [root@hypervisor ~]# oc get co image-registry
NAME             VERSION   AVAILABLE   PROGRESSING   DEGRADED   SINCE   MESSAGE
image-registry   4.16.8    True        False         False      61s

Install Node.js application on Tenant cluster

  1. From the left navigation pane, click Administrator  Developer.

  2. You will be redirected to developer view.

    Click on text create a Project.

  3. In the create project window, provide the project details.

    Name: sample.

    Display name: sample.

    Description: This is sample project.

    Click Create to create the project.

  4. In the project view, to add the sample application click on text Add page.

  5. In the add page window, click on text View all samples.

  6. In the search section, search for basic node.

    Select the Basic Node.js application from samples.

  7. In Basic Node.js application building window, keep all options as is and click Create to create the application.

  8. In topology view, you will see application is created.

    Click on the circle to view application overview.

  9. In resources tab, notice the application is building.

    If you click the URL from routes section, it will open the application page in new tab.

  10. Click Advanced…​ to proceed.

  11. Click Accept the Risk and Continue to accept the risk and proceed.

  12. As application is still building stage, you will notice the message as Application is not available in page.

  13. In a minute, you will notice the application is in running state.

  14. Refresh the application page to see the default page message.

Test the High Availability and Resilience

  1. Find out the where the application pod is running.

    Sample output:
    [root@hypervisor ~]# oc get pods -n sample -o wide
NAME                            READY   STATUS      RESTARTS   AGE     IP            NODE                   NOMINATED NODE   READINESS GATES
nodejs-basic-1-build            0/1     Completed   0          2m51s   10.130.0.63   tcn2.lab.example.com   <none>           <none>
nodejs-basic-6d55569c9c-gps2d   1/1     Running     0          2m51s   10.130.0.62   tcn2.lab.example.com   <none>           <none>

  2. In this case, the application pod is running on the tcn2.lab.example.com node.

    This means tcn2.lab.example.com node is running on sno2 Infrastructure cluster.

  3. To test high availability and resilience, shutdown the sno2 Infrastructure cluster and to do this you need to shutdown the sno2 VM.

    Ensure sno2 VM is in shutoff state.

  4. If you refresh the application page, you will notice failure message again.

  5. The pod-eviction-timeout and node-monitor-grace-period parameters have the default value of 5m and 40s respectively, which means by default, it takes 5m40s to get pod eviction process triggered since the last status update from the node.

    After five minutes i.e. eviction timeout, you will notice the application is successfully migrated to tcn1.lab.example.com node.

    Sample output:
    [root@hypervisor ~]# oc get pods -n sample -o wide
NAME                            READY   STATUS        RESTARTS   AGE   IP             NODE                   NOMINATED NODE   READINESS GATES
nodejs-basic-6d55569c9c-gps2d   1/1     Terminating   0          23m   10.130.0.62    tcn2.lab.example.com   <none>           <none>
nodejs-basic-6d55569c9c-tdkbm   1/1     Running       0          13m   10.128.1.115   tcn1.lab.example.com   <none>           <none>
    Eviction timeout - How to modify the pod reschedule timeout in OpenShift 4?

  6. Refresh the application page again to view the default message again.

  7. This test shows, even though one infrastructure node is down; application automatically migrate to other infrastructure node.
